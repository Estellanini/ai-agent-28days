# Glossary

## Agent

- 我的定义：能自主决策、调用工具的带流程的智能体。
- 解决什么：让模型不只是"问一句答一句"，而是能自己拆任务、决定调用哪些工具、一步步把事做完，直到达成目标。"流程"就是它自己驱动的那条执行路径。
- 不解决什么：不保证一定做对——模型可能判断错、工具可能失败；它本身也不创造新能力，能力全部来自底层模型 + 工具。
- 最小例子：你说"把这段代码格式化并提交"，它自己读文件 → 改格式 → 跑检查 → git commit，每一步都由它自己决定下一步。

## Agent Loop

- 我的定义：思考 → 行动 → 看结果 → 再思考，反复迭代直到结束。
- 解决什么：让 agent 能"反复"跑——拿到当前状态 → 模型决定下一步（调工具 or 结束）→ 执行 → 把结果喂回模型 → 再决定……一圈圈转，直到模型说"完成"。Loop 的灵魂是"重复"，不是一次性的输入输出。
- 不解决什么：不判断每一步对不对，只是机械地跑圈；模型陷入死循环或走错路时，loop 本身不会纠偏，要靠外面的 guardrail / eval 兜。
- 最小例子：搜问题 → 看结果不够 → 换关键词再搜 → 够了 → 总结回答。"搜→看→再搜→停"就是好几圈 loop。

## Tool

- 我的定义：工具，赋予智能体一些第三方能力。
- 解决什么：让模型能做它本身做不到的事——读文件、执行代码、查数据库、发请求、改文件。模型只有"语言"，工具给它"手脚"，而且会产生真实的副作用。
- 不解决什么：不提升模型的判断力；工具只负责"你叫我干嘛我干嘛"，该不该用、用得对不对仍由模型（或人）决定；它也不"理解"自己在干嘛。
- 最小例子：Read 工具——模型说"读 /tmp/a.txt"，工具真的去文件系统读出内容返回给模型。

## Tool Schema

- 我的定义：工具的"使用说明书"——告诉模型这个工具叫什么、干什么、要哪些参数、参数是什么类型。
- 解决什么：让模型知道"什么时候该用它、怎么传参"，从而能正确生成工具调用。没有 schema，模型根本不知道有这个工具，更别说怎么用。
- 不解决什么：不保证工具内部逻辑对——它只是接口契约（签名），工具真的执行起来对不对是另一回事；也不保证模型一定会用对。
- 最小例子：search 工具的 schema 写明 name="search"、description="全文搜索代码"、parameters={query: string, max_results: int}，模型照着生成调用。

## Skill

- 我的定义：把技能或者流程标准化。
- 解决什么：把一类任务的"最佳做法"写成提示词 / 流程文档，模型遇到这类任务时按它走。它是"脑子里的方法论"而非"手脚"，而且通常是按需加载（用到才读，省 context）。
- 不解决什么：不直接执行代码、不自己产生副作用——它只影响模型"怎么想、怎么规划"。和 tool 的关键区别：skill 是说明书，tool 是扳手。
- 最小例子：code-review skill 在遇到"审查代码"时被加载，告诉模型按一份清单去 review；它本身不改代码，是模型按 skill 给的方法去调工具完成。

## Harness

- 我的定义：跑 agent 的"外壳 / 底座"——把模型、工具、权限、内存、trace 这些零件装起来、串起来的运行框架。
- 解决什么：让 agent 能在一个可控环境里安全跑起来——管工作目录、决定暴露哪些工具、卡权限、记 trace、出错能恢复。开发者不用每次从零拼装。
- 不解决什么：不提升模型本身的能力，不管业务逻辑对不对，它也不是"模型"——它只负责把零件组织好、把流程跑顺。
- 最小例子：Claude Code 就是一个 harness：把 Claude 模型 + 文件 / 命令工具 + 权限确认 + 会话记忆，包成一个能用的 CLI。

## MCP

- 我的定义：模型上下文协议，一个标准协议，能够让第三方能力都丝滑地接入智能体。
- 解决什么：让工具 / 资源 / 提示能"统一插口"地接进任何 agent——以前每个 agent 要为每个工具单独写适配，现在写一次 MCP server，谁都能用（像 USB 之于设备）。
- 不解决什么：不规定业务逻辑，也不保证安全——MCP 只是"传输和格式标准"，工具接进来对不对、危不危险，还得靠人和 harness 把关。
- 最小例子：一个 MCP server 把公司数据库暴露成标准工具，Claude Code、Cursor、自己的脚本都能直接接，不必各写一套适配。

## Memory

- 我的定义：让 agent 能"记住"东西——跨步骤或跨对话地保留信息（用户偏好、过往决策、项目背景）。
- 解决什么：让 agent 不必每次从零开始，能基于历史和上下文做更准的决策，解决"上一轮说的事这轮就忘了"。
- 不解决什么：不是模型自带的参数记忆，也不是无限大的——要取舍存什么、怎么检索，存错了反而误导；它也不等于"知识"。
- 最小例子：用户说"我喜欢 tabs 缩进"，agent 存进 memory，下次写代码默认用 tabs，而不是每次重问。

## Trace

- 我的定义：agent 跑一圈留下的"完整日志"——每一步想了什么、调了哪个工具、传了什么参数、返回了什么、花了多少 token。
- 解决什么：把 agent 的黑盒过程变可观测、可复盘、可调试；出问题时能精确定位是哪一步、哪个工具、哪个决策出错。
- 不解决什么：不自动修复问题——它只是"记录"；它也不等于 eval（eval 是评判，trace 是被评判的原材料）。
- 最小例子：一次任务的 trace 里能看到"调了 3 次 search、第 2 次传了空 query、模型在那圈决定换关键词"，这帮你定位结果为什么不好。

## Eval

- 我的定义：给 agent 的表现打分 / 把关的一套方法——用一批测试用例看它做得对不对、稳不稳、安不安全。
- 解决什么：把"感觉还行"变成"可衡量"——用数据判断改动是好是坏，避免靠感觉调 prompt 越调越乱。
- 不解决什么：测试集有限，不能覆盖所有情况；eval 高分不等于线上一定好；它只评判"输出质量"，不直接帮你修 bug。
- 最小例子：准备 50 个任务 + 每个的期望结果 / 检查点，agent 跑完按 rubric 打分看通过率；改完 prompt 再跑一次，对比分数涨没涨。
