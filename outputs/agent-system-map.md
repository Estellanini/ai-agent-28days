# Agent System Map

```text
User
  -> CLI / App / API
  -> Agent Harness
      -> Model
      -> Tool Registry
      -> State / Memory
      -> Policy / Approval
      -> Trace / Eval
      -> Runtime / Sandbox
```

## 箭头解释

每条：谁调用谁 → 传什么 → 失败会在哪里发生。

- **User -> CLI / App / API**：用户在终端 / 网页 / API 客户端敲下自然语言请求。
  - 传什么：用户的目标（一句话需求）。
  - 失败：意图本身就模糊或错（"把那个文件改一下"——哪个？），脏数据从源头就进来了。

- **CLI / App / API -> Agent Harness**：前端把请求交给 harness 这个"底座"。
  - 传什么：用户输入 + 会话上下文（session id、配置、工作目录）。
  - 失败：上下文丢失或配置没传对（该给的权限没给、工作目录指错），harness 拿到的是残缺任务。

- **Agent Harness -> Model**：harness 把组装好的 prompt 发给模型，拿回下一步决策。
  - 传什么：system prompt + 可用工具列表（schema）+ 历史消息。
  - 失败：模型输出不可解析（工具调用格式错）、幻觉、调用根本不存在的工具；或限流 / 超时。

- **Agent Harness -> Tool Registry**：harness 维护一份"有哪些工具可用"的注册表，按策略决定暴露哪些给模型。
  - 传什么：工具的 schema（名字、参数、描述）。
  - 失败：暴露了不该暴露的危险工具（安全风险）；或 schema 写得差，导致模型用错工具 / 传错参。

- **Agent Harness -> State / Memory**：harness 在每一步读写状态和记忆。
  - 传什么：当前会话历史、用户偏好、项目背景；以及每一轮模型的输出。
  - 失败：上下文超长撑爆 context window、记忆检索召回错误的旧信息、多步之间状态不一致。

- **Agent Harness -> Policy / Approval**：执行工具前，harness 先过一遍策略 / 审批闸门。
  - 传什么：即将执行的动作（工具调用 + 参数）+ 权限规则。
  - 失败：策略太松 → 危险动作直接放行（如 `rm -rf`）；太严 → 每步都要人确认，体验卡死；审批被绕过。

- **Agent Harness -> Trace / Eval**：harness 把每一步执行情况落成 trace，事后用 eval 评判。
  - 传什么：每步的输入输出、工具调用、token、耗时。
  - 失败：trace 没记全 / 记错 → 出问题无法复盘；eval 覆盖不全 → 漏掉真实故障场景，高分却线上翻车。

- **Agent Harness -> Runtime / Sandbox**：harness 在沙箱 / 运行环境里真正执行工具的副作用。
  - 传什么：要执行的命令 / 文件操作 / 网络请求。
  - 失败：沙箱隔离失效 → 工具越权访问宿主系统；命令执行失败（exit 非 0）；CPU / 磁盘 / 网络资源耗尽。

## 失败点

按层归总典型问题（和上面的箭头失败点呼应）。

- **模型层**：输出格式错（工具调用解析失败）、幻觉、调用不存在的工具、限流 / 超时。
- **工具层**：工具执行失败（exit code 非零）、参数传错、暴露了危险工具、schema 写得太差导致模型误用。
- **状态层**：上下文超长撑爆窗口、记忆召回错误的旧信息、多步执行状态不一致。
- **权限层**：策略过松导致危险操作被放行、过严导致流程卡死、审批闸门被绕过。
- **运行环境层**：沙箱逃逸（工具越权访问宿主）、资源耗尽（CPU / 磁盘 / 网络）、底层命令本身失败。

## 一句话总结

Harness 是中枢：对上接用户、对下接模型和真实世界，中间用工具 / 状态 / 权限 / trace 把"模型的语言能力"转成"可控、可观测、可恢复的真实操作"。任何一层都可能失败，所以架构的每个箭头都要有"失败会在哪里发生"的预案。
